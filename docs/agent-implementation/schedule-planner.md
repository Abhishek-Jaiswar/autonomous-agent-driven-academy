# Schedule Planner Agent Implementation Specification

## 1. Overview
The **Schedule Planner Agent** takes the structured syllabus (phases, modules, lessons) generated by the Curriculum Architect and maps it chronologically across the student's timeline (e.g., a 21-day calendar). It ensures the student has a steady, achievable daily study load.

* **Execution Context**: BullMQ Job Worker (`generate-curriculum` job).
* **Trigger Event**: Executes immediately after the Curriculum Architect finishes writing the syllabus transaction.
* **Database Target**: `Lesson` model status fields.

---

## 2. Scheduling & Distribution Logic
1. **Lessons Extraction**: Queries the database to retrieve all lessons created for the `curriculumId`, ordered by phase order, module order, and lesson order.
2. **Day Assignment**:
   - Calculates the average lesson load per day: `Load = Total Lessons / durationDays`.
   - Distributes lessons sequentially across days (Day 1, Day 2 ... Day N).
   - For example: if there are 14 lessons and a 7-day timeline, assigns exactly 2 lessons per day.
3. **Lesson Status Activation**:
   - Updates all lessons in the database.
   - The first lesson (Day 1, Lesson 1) is marked as `UNLOCKED`.
   - All subsequent lessons are marked as `LOCKED` by default.
4. **Log Updates**: Writes an `AgentLog` checkpoint: `"Schedule Planner mapped 14 lessons across 7 days. Day 1 is unlocked."`
5. **WebSocket Dispatcher**: Emits a `curriculum-ready` event containing the compiled syllabus calendar, prompting the frontend to transition from the Counselor screen to the Curriculum Map workspace.

---

## 3. Database Updates
```typescript
// Example status activation
const lessons = await db.lesson.findMany({
  where: { module: { phase: { curriculumId } } },
  orderBy: [
    { module: { phase: { order: "asc" } } },
    { module: { order: "asc" } },
    { order: "asc" }
  ]
});

// Update the first lesson status
await db.lesson.update({
  where: { id: lessons[0].id },
  data: { status: "UNLOCKED" }
});
```
